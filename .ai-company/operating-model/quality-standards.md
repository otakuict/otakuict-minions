# Quality Standards

## Definition of Done

Before calling work done, confirm:

- the requested output exists
- acceptance criteria are met
- important files or artifacts are listed
- obvious risks are addressed
- memory is updated when relevant

## Verification

Use verification proportional to risk:

- Documents: check structure and filenames.
- Code: run focused tests or validation where practical.
- Data: check schema, row counts, nulls, ranges, and assumptions.
- Marketing: check audience, offer, clarity, channel fit, and next action.
- Product: check user, problem, scope, edge cases, and acceptance criteria.

## Review Bias

Look for:

- unclear ownership
- missing acceptance criteria
- unsupported assumptions
- stale memory
- skipped validation
- overbuilt plans
